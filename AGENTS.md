# AGENTS.md

## 编码要求

本项目所有文本内容使用 UTF-8 编码。

## 项目名称

管理学原理知识图谱 / 思维导图项目

## 项目背景

本项目服务于本科课程“管理学原理”的课程作业。作业要求是依据管理学原理课程内容、管理学学科知识，并结合本人理解与个人经验，设计一份管理学知识图谱或思维导图，体现管理学核心知识点及其相互联系。

## 当前对话共识

本项目已经确定以下工作共识：

1. 最终渲染工具采用 XMind。
2. AI 协作阶段不直接生成 JPG/PNG 等不可编辑图片。
3. AI 主要维护 Markdown 源文件，项目主人通过 Markdown 手动导入 XMind。
4. XMind 用于最终可视化编辑、布局调整、关系线补充、颜色样式设计与导出。
5. 项目重点不是简单罗列教材目录，而是先打磨管理学知识体系，再寻找主线和知识点之间的联系，最后完成绘图。

推荐工作流：

```text
资料校准 -> 知识体系打磨 -> 主线提炼 -> 关系建模 -> Markdown 成稿 -> XMind 渲染
```

## 项目目标

生成一份能够体现“我理解中的管理学原理”的知识图谱。图谱应当做到：

1. 覆盖本科管理学原理课程的主流核心知识点。
2. 参考主流教材框架，避免遗漏重要概念。
3. 不机械照搬教材目录，而是重构为清晰的个人化知识体系。
4. 明确展示知识点之间的因果、支撑、反馈、制约、演化等关系。
5. 保留可编辑源文件，便于反复修改和导入 XMind。

## 知识体系方向

管理学原理知识图谱的中心主题建议为：

```text
管理学原理：组织如何在变化环境中协调人与资源以实现目标
```

知识体系可以围绕以下模块展开：

```text
管理基础
管理思想发展
组织环境
决策与战略
计划
组织
领导
激励
沟通
控制
组织文化
变革与创新
个人理解与实践案例
```

其中，“计划、组织、领导、控制”是传统管理职能主干；“环境、决策、战略、变革”用于体现动态性；“激励、沟通、文化、团队”用于体现人的因素。

## 推荐主线

本项目推荐采用以下主线串联知识图谱：

```text
环境变化 -> 管理决策 -> 目标与计划 -> 资源配置 -> 组织协同 -> 领导与激励 -> 控制反馈 -> 变革创新
```

也可以在图中保留经典管理职能闭环：

```text
计划 -> 组织 -> 领导 -> 控制 -> 反馈修正计划
```

## 关系建模要求

本项目不仅要画层级结构，还要体现知识点之间的联系。应重点表达以下关系：


## XMind 工作流约定

XMind 作为最终渲染和人工编辑工具。AI 阶段应优先生成适合 XMind 导入的 Markdown，而不是直接生成图片。依据 XMind 官方导入规则，Markdown 的标题会映射为节点层级，列表会按缩进映射为子主题，第一行会被用作中心主题。因此 `xmind-import.md` 应写成“可导入大纲”，而不是普通文章。

### 格式选型结论

本项目最适合 AI 调整、并能通过 XMind 导入且导入后操作最少的格式是：

```text
受约束的 XMind-import Markdown
```

也就是说，项目源稿不是普通 Markdown 文章，而是专门为 XMind 导入优化的标题层级大纲。

格式优先级如下：

```text
首选：受约束 Markdown
可选：OPML
不推荐作为 AI 源稿：TextBundle、DOCX、Mermaid、Graphviz、直接编辑 .xmind
```

选型理由：

1. Markdown 对 AI 最友好，便于反复改写、审阅、diff 和版本管理。
2. XMind 官方明确支持 Markdown 导入，且规则清楚：标题映射节点层级，列表映射子主题。
3. OPML 适合表达纯树状大纲，但本质是 XML；AI 反复编辑时更容易出现转义、标签闭合或层级错误，人工阅读也不如 Markdown 直观。
4. TextBundle 主要适合打包 Markdown 与图片素材，本项目目前不依赖大量图片，因此没有必要增加复杂度。
5. DOCX 适合提交文档，不适合作为 AI 长期维护的结构化源文件。
6. Mermaid 和 Graphviz 适合生成图形或网络关系图，但不能很好地作为 XMind 的可编辑导入源。
7. `.xmind` 是最终编辑文件，不应作为 AI 阶段的主源稿；除非后续接入可靠的 XMind MCP 或原生编辑工具。

因此，后续默认维护：

```text
outline.md       用于知识体系打磨
relations.md     用于横向关系建模
personal_view.md 用于个人理解与案例
xmind-import.md  用于最终导入 XMind
```

`xmind-import.md` 必须保持“标题层级优先、短列表补充”的形式，以减少导入后的人工整理成本。

### 导入文件与后处理文件分工

`xmind-import.md` 是唯一应该直接导入 XMind 的文件。它必须写成严格的 XMind-import Markdown，只保留：

```text
中心主题
主分支
子分支
短语化节点
少量关键联系提示
```

`relations.md` 不作为主要导入文件。它是导入 XMind 后的操作指南，负责说明：

```text
哪些节点之间需要添加 Relationship
关系线文字写什么
哪些模块适合加 Boundary
哪些分组适合加 Summary
哪些层次或模块适合用颜色区分
哪些节点导入后需要折叠、移动、精简或突出
```

导入后操作时，优先根据 `relations.md` 在 XMind 中补充跨分支联系，例如：

```text
计划 --提供标准--> 控制
控制 --反馈修正--> 计划
环境变化 --影响--> 决策
战略 --决定--> 组织结构
激励 --影响--> 工作绩效
```

### XMind 导入 Markdown 规范

`xmind-import.md` 应遵循：

1. 文件扩展名必须为 `.md`，文本编码使用 UTF-8。
2. 第一行必须是中心主题，推荐写作 `# 管理学原理知识图谱`，不要在文件开头放 YAML front matter、说明文字、空行或目录。
3. 使用 `#` 到 `######` 表示节点层级，最多 6 层；超过 6 层的内容应压缩或拆分到其他文件。
4. 主干结构优先使用标题层级，细节节点可使用缩进列表。
5. 列表项使用 `-`，每一级缩进保持一致；不要混用无序列表、有序列表和不稳定缩进。
6. 每个节点尽量是短语或短句，避免把解释性长段落直接放进导入稿。
7. 关系说明可以作为“关键联系”分支导入，但真正的跨分支关系线应在 XMind 中手动添加。
8. `xmind-import.md` 中避免使用表格、脚注、复杂引用块、Mermaid、HTML、大段代码块等非导图核心内容。

推荐导入格式：

```markdown
# 管理学原理知识图谱

## 管理基础

### 管理的含义

- 协调人与资源
- 实现组织目标
- 兼顾效率与效果

### 管理者角色

- 人际角色
- 信息角色
- 决策角色

## 管理过程主线

### 计划

- 目标设定
- 环境分析
- 方案选择

### 组织

- 资源配置
- 结构设计
- 权责划分

### 领导

- 激励
- 沟通
- 团队

### 控制

- 标准设定
- 绩效衡量
- 反馈修正

## 关键联系

### 计划与控制

- 计划提供控制标准
- 控制结果反馈修正计划
```

### Markdown 与 XMind 的分工

Markdown 负责：

```text
中心主题
层级结构
节点命名
简要说明
待添加关系线清单
```

XMind 负责：

```text
布局方向
分支颜色
关系线
边界
概要
标签
图标与标记
最终导出
```

导入 XMind 后，应重点手动处理：

```text
用 Relationship 添加跨分支联系，并给关系线添加文字说明
用 Boundary 标出同一模块或同一逻辑区域
用 Summary 概括同一组节点的共同含义
用颜色区分基础层、职能层、行为层、环境层、应用层
检查节点是否过长、层级是否过深、画面是否拥挤
最终导出格式
```

建议导入路径：

```text
XMind 桌面端：File -> Import -> Markdown (.md)
XMind Web：All Maps -> + Create New -> Import -> Upload Files / Paste Text
```

## 关系类型图例

图谱中的横向联系应尽量区分关系类型。这样导入 XMind 后，关系线不只是“连起来”，而是能表达知识点之间的逻辑。

建议使用以下关系类型：

```text
因果关系：A 导致或影响 B
示例：环境变化 -> 管理决策

支撑关系：A 为 B 提供条件
示例：沟通反馈 -> 组织协同

约束关系：A 限制或规范 B 的选择
示例：社会责任 -> 战略选择

反馈关系：A 的结果反过来修正 B
示例：控制反馈 -> 目标与计划

转化关系：A 经过管理活动转化为 B
示例：决策过程 -> 目标与计划

权衡关系：两个目标之间存在张力，需要管理者平衡
示例：标准控制 <-> 创新弹性
```

在 `relations.md` 中记录关系线时，优先写清：

```text
起点节点
关系类型
关系线文字
终点节点
简短含义
```

## 构图叙事原则

最终 XMind 构图建议采用：

```text
中心主题 + 动态主轴 + 职能闭环 + 外围支撑
```

推荐构图逻辑：

```text
中心：
管理学原理知识图谱

主轴：
环境变化 -> 决策与战略 -> 计划 -> 组织 -> 领导与激励 -> 控制反馈 -> 变革创新

内圈：
计划、组织、领导、控制
用于体现经典管理职能闭环

外圈：
管理思想、组织环境、文化、伦理、社会责任、个人案例
用于体现理论来源、环境约束和个人理解

横向关系线：
用于表现跨模块联系，不把所有关系塞进树状层级
```

构图时的优先级：

```text
第一优先：让主线一眼可见
第二优先：保留核心知识点完整性
第三优先：补充理论细节和个人案例
第四优先：调整颜色、边界、图标和视觉风格
```

如果画面拥挤，优先折叠理论细节分支，而不是删除主线、闭环和关键关系线。

## 节点粒度规范

`xmind-import.md` 中的节点应保持短语化、层级清楚、便于导入 XMind 后继续编辑。

建议遵循：

```text
一级分支：大模块
二级分支：核心概念、管理职能或关键问题
三级分支：理论、工具、过程或机制
列表项：短语化细节
```

节点表达要求：

```text
单个节点尽量不超过 15 个汉字
同一层级尽量保持语法一致
优先使用名词短语或动宾短语
避免把判断句、解释句、长段案例直接放入导入稿
长解释放入 outline.md
个人案例叙述放入 personal_view.md
跨分支关系放入 relations.md
```

可接受节点示例：

```text
环境扫描
战略选择
目标体系
组织结构
沟通反馈
文化惯性
控制反馈
```

不推荐节点示例：

```text
由于外部环境不断变化，所以管理者必须根据具体情境不断调整计划
我认为管理最重要的是让每个人都知道自己要做什么并保持沟通
```

## 个人化表达提取模板

加入个人案例时，先在 `personal_view.md` 中展开，再提炼为 `xmind-import.md` 中的短节点。

推荐模板：

```text
场景：我在哪个组织、任务或经历中观察到这个问题
问题：出现了什么管理矛盾或协同困难
理论：可以对应哪个管理学概念
做法：当时如何计划、组织、沟通、控制或调整
反思：这个案例说明我如何理解管理
```

压缩进导图时，只保留最有解释力的短节点，例如：

```text
分工清晰减少返工
进度反馈修正计划
沟通滞后增加协同成本
共同目标提升投入感
复盘推动流程改进
```

个人化表达不应成为单独孤立的装饰分支。导入 XMind 后，应尽量把个人案例与以下模块建立关系线：

```text
管理基础
计划
组织
领导与人的因素
控制与反馈
变革与创新
```

## 最终成稿验收标准

最终提交前，应从“内容完整性、结构表达、个人理解、XMind 可读性”四个方面检查。

验收标准：

```text
一眼能看出中心主题和主线
主分支数量适中，画面不拥挤
经典职能闭环清晰可见
至少 8 条核心横向关系已在 XMind 中补充
个人理解不是附属装饰，而能连接到管理理论
理论名称准确，节点表达简洁
关系线文字短而清楚
颜色、边界和概要服务于理解，而不是单纯装饰
导图既能体现课程知识，也能体现自己的重构和理解
```

若最终图谱显得过于复杂，优先保留：

```text
中心主题
动态主线
经典职能闭环
关键横向关系
个人化案例
```

理论细节可以折叠、放入备注，或保留在 `outline.md` 中作为解释材料。

## Git 提交信息规范

本项目提交信息采用 Conventional Commits 风格，并结合常见 Git 社区写法。

推荐格式：

```text
<type>(<scope>): <subject>
```

其中：

```text
type    表示变更类型
scope   可选，表示影响范围
subject 使用简短中文描述本次变更
```

常用类型：

```text
feat      新增功能、内容能力或主要产物
fix       修复错误
docs      文档、说明、规范、文字内容更新
style     格式、排版、标点、空白等不影响内容语义的调整
refactor  重构结构但不改变功能或内容意图
test      测试相关
chore     构建、依赖、配置、杂项维护
```

本项目中，修改 `AGENTS.md`、`README.md`、说明文档、参考资料、工作流规范时，优先使用 `docs:`。

提交信息示例：

```text
docs(agents): 完善 XMind 导入工作流规范
docs(outline): 补充管理学知识体系主线
feat(mindmap): 新增 XMind 导入稿初版
fix(relations): 修正控制反馈关系描述
chore(repo): 添加导出目录占位说明
```

提交信息要求：

1. 首行使用 `type(scope): subject`；scope 可省略。
2. 首行尽量控制在 50 个字符左右，最多不超过 72 个字符。
3. subject 使用祈使式或结果导向表达，不写句号。
4. 如需解释原因，在首行后空一行，再写正文。
5. 正文说明“为什么改”和“影响什么”，不要重复代码或文件 diff。
6. 正文每行尽量不超过 72 个字符。
7. 每次提交尽量保持单一主题，避免把内容规划、导图源稿、导出文件混在一个提交里。

参考规范：

```text
Conventional Commits 1.0.0
Angular Commit Message Guidelines
How to Write a Git Commit Message
```

## 后续协作原则

后续 AI 协作应遵循：

1. 先完善知识体系，再讨论视觉效果。
2. 先保证内容准确完整，再压缩成适合图谱的短语。
3. 每个核心节点尽量使用简洁短语，避免长段落直接进入 XMind。
4. 横向关系单独整理，避免主图过度拥挤。
5. 重要修改优先更新 Markdown 源文件，再同步到 XMind。
6. 若图谱过大，可拆分为多个 XMind sheet 或多个 Markdown 分图。

## 不推荐做法

本项目不推荐：

```text
直接让 AI 生成不可编辑 JPG/PNG
只按教材目录机械罗列章节
一开始就追求视觉美化
把所有解释性文字塞进节点
忽略知识点之间的横向联系
只保存 XMind 文件而不保留 Markdown 源稿
```
